## Project Overview

As part of the **SAE4.DEVCLOUD.01** project at IUT de Colmar, I contributed to the design and implementation of a complete **banking management interface** based on a **microservices architecture**. This project involved developing a full-stack application with account management, user authentication, transaction processing, and validation workflows for bank agents. The application was deployed in a virtualized environment using Docker and Docker Compose, demonstrating modern cloud-native development practices.

The project required close collaboration in a two-person team, where I was responsible for the frontend development while my teammate handled the backend API. This division of responsibilities allowed us to work in parallel while maintaining clear interfaces between our components.

## Technical Architecture

### Microservices Architecture

The application follows a microservices architecture pattern where each component is independently deployable and scalable:

- **Backend Service (FastAPI)**: RESTful API service handling business logic, data persistence, and authentication.
- **Frontend Service (Django)**: Web interface for clients and bank agents to interact with the banking system.
- **Database Service (MySQL)**: Centralized data storage for users, accounts, and transactions.
- **Adminer Service**: Database administration interface for development and debugging.

This architecture validates microservices competency through several key characteristics. Each service operates in its own Docker container, enabling independent scaling based on specific service demands. Communication between services occurs through well-defined REST APIs, ensuring loose coupling and service autonomy. This modular structure facilitates future expansion while allowing each component to evolve independently.

### Backend Architecture (FastAPI)

The backend was developed using FastAPI, a modern Python web framework chosen for its high performance and native asynchronous capabilities.

**Technical Stack:**
- **Framework**: FastAPI (Async/Await)
- **ORM**: Tortoise ORM
- **Database**: MySQL 9.3
- **Security**: Passlib (SHA256-Crypt) & HTTP Basic Auth
- **Tools**: Schwifty (IBAN), PDM (Package Management)

The implementation leverages `async/await` throughout to handle concurrent requests efficiently. Database interactions are handled through Tortoise ORM, providing a clean abstraction over the MySQL store. Security is integrated at the core, with encrypted password storage and IBAN validation ensuring data integrity.

The service manages the entire banking lifecycle: from user registration and automatic IBAN generation to complex transaction processing. A sophisticated validation workflow requires bank agents to approve withdrawals and transfers, while custom middleware maintains detailed audit trails of every API interaction.

### Frontend Architecture (Django)

I took the lead on developing the Django frontend, focusing on creating a robust and user-friendly portal for both clients and bank staff.

**Key Features:**
- **Role-Based Access**: Dedicated portals for clients and bank agents.
- **Real-Time Integration**: Seamless communication with the FastAPI backend.
- **Security**: Secure session management with HTTP Basic Auth integration.
- **User Experience**: Intuitive dashboards for transaction management and account validation.

The frontend acts as a bridge between the complex backend logic and the end-user. Using Django's template engine, I built dynamic views that consume the REST API. I focused heavily on error handling and state management to ensure that users receive immediate, clear feedback during sensitive operations like money transfers or account approvals.

## Infrastructure and Virtualization

### Docker Containerization

The architecture is built on a container-first approach, where each service operates within its own isolated environment. This design ensures that frontend, backend, and database components remain decoupled, preventing dependency conflicts and simplifying the deployment lifecycle.

- **Frontend Container**: Serves the Django application on port 80.
- **Backend Container**: Runs the FastAPI application on port 8001.
- **Database Container**: Manages a MySQL 9.3 instance on port 3306.
- **Adminer Container**: Provides a lightweight database administration tool on port 8002.

By virtualizing the entire stack, I ensured that the development environment is an exact replica of the production target. This isolation validates my competency in virtualization; containers can be versioned, scaled independently, and deployed consistently across any infrastructure without "works on my machine" issues.

### Docker Compose & Network Automation

To orchestrate these services, I utilized **Docker Compose** to define the infrastructure as code. This allows for an automated and reproducible setup where service dependencies are strictly managed through health checksâ€”ensuring the frontend only starts once the backend is fully operational, and the backend only after the database is ready.

The networking is strategically segmented into two tiers:
- **`front-back` Network**: A public-facing segment connecting the user interface and API.
- **`internals` Network**: A private, isolated segment for secure backend-to-database communication.

This dual-network approach minimizes the attack surface by ensuring sensitive database traffic is never exposed to the external network. Service discovery is handled automatically via Docker's internal DNS, allowing components to communicate using service names instead of volatile IP addresses.

## Development and Deployment Practices

### Project Planning and API Integration

A critical part of this project was the coordination between frontend and backend. I used a **Gantt chart** to map out dependencies and ensure that API contracts were established before any code was written. This systematic approach allowed for parallel development, where the Django frontend could be built against defined FastAPI endpoints without waiting for the full backend logic to be complete.

The integration layer I developed in Django features:
- **HTTP Basic Auth**: Secured communication using a standard authentication layer.
- **Reusable API Clients**: Modular functions designed to handle various REST methods (GET, POST, DELETE).
- **Graceful Error Handling**: Robust mechanisms to parse API responses and provide meaningful user feedback, even during service interruptions.

### Security & DevOps

Security was integrated as a primary requirement rather than an afterthought. We implemented industry-standard **SHA256-Crypt** hashing for passwords and strictly enforced the **principle of least privilege** through multi-stage validation workflows. From a DevOps perspective, the entire pipeline is automated: a single `docker compose up` command provisions the network, configures volumes for persistent data, and starts the synchronized services, demonstrating a professional-grade deployment workflow.

## Technical Challenges and Reflections

Developing a distributed banking system presented several non-trivial hurdles. **Service synchronization** was a major focus; ensuring that the database was fully healthy before the backend attempted migrations required fine-tuning Docker health checks. Another challenge was the **parallel development** workflow. By strictly adhering to API contracts established in our planning phase, we were able to build the Django frontend and FastAPI backend simultaneously, using mock responses to validate UI behavior before the final integration.

These experiences reinforced several key professional outcomes:
- **Cloud Solutions**: Validated the ability to propose and implement scalable architectures.
- **DevOps Culture**: Bridged the gap between development and operations through automated deployment.
